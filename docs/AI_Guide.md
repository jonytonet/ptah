## Project Context

Before any prompt, give the agent the necessary context. Ptah has a specific architecture that the agent needs to know:

```
Package:   jonytonet/ptah
Laravel:   12.x  |  PHP 8.3  |  Livewire 4  |  Tailwind v4  |  Alpine.js 3
Icons:     Boxicons 2.1.4 + FontAwesome 6.7.2 (via CDN — never inline SVG)
Dark mode: `.ptah-dark` class on the root element — CSS centralised in forge-dashboard-layout
Locale:    PTAH_LOCALE=en (default) | pt_BR — UI strings via `__('ptah::ui.*')`
Tests:     Orchestra Testbench + PHPUnit 11 + SQLite :memory:
Audit:     HasAuditFields trait (Ptah\Traits\HasAuditFields) — every generated model includes
           created_by/updated_by/deleted_by filled automatically via Eloquent events;
           BaseCrud explicitly injects them in save()/deleteRecord(); bulkDelete() uses each()
```

> **Tip:** run `php artisan ptah:install --boost` to install [Laravel Boost](https://laravel.com/docs/12.x/boost)
> automatically. With Boost active, agents (Copilot, Claude Code, Cursor) receive Ptah guidelines and
> skills every session — no need to paste context manually into prompts.

**Layers generated by `ptah:forge`:**

```
app/
├── Http/Requests/{Entity}/Store{Entity}Request.php
├── Http/Requests/{Entity}/Update{Entity}Request.php
├── Http/Resources/{Entity}/{Entity}Resource.php
├── DTO/{Entity}/{Entity}DTO.php
├── Contracts/
│   ├── Repositories/{Entity}RepositoryContract.php
│   └── Services/{Entity}ServiceContract.php
├── Repositories/{Entity}Repository.php
├── Services/{Entity}Service.php
└── Models/{Entity}.php

database/migrations/YYYY_MM_DD_create_{entities}_table.php
resources/views/{entity}/index.blade.php           ← uses BaseCrud
```

## Anatomy of a Good Prompt

An effective prompt for code generation with Ptah should contain:

```
[CONTEXT]       → What is the entity's role in the system?
[FIELDS]        → What fields exist and their types?
[RULES]         → Validations, uniqueness, relationships.
[INTEGRATION]   → Which module does it use? (company, permissions, menu?)
[PATTERN]       → Architecture reminder (SOLID, BaseCrud, Ptah)
```

## Prompt: Create a New Entity

Use this template when you want to create an entity from scratch with Ptah scaffolding.

```
You are working on the jonytonet/ptah package (Laravel 12, Livewire 4,
Tailwind v4). Use the ptah:forge command to generate the entity.

ENTITY: Product

FIELDS:
  - name:string         required, max:255
  - sku:string          required, unique, max:50
  - price:decimal(10,2) required, min:0
  - stock:integer       default 0, min:0
  - category_id:FK      belongsTo Category relationship
  - is_active:boolean   default true
  - description:text    nullable

BUSINESS RULES:
  - SKU cannot be changed after the first order
  - Inactive products must not appear in the public catalogue
  - soft_delete enabled

GENERATE:
  1. The ptah:forge command with the fields above
  2. Complete migration with indexes on sku and category_id
  3. Model with casts, fillable and `active()` scope
  4. DTO, Repository, Service and Contracts following SOLID
  5. Validation rules in StoreProductRequest / UpdateProductRequest
  6. [Optional] Add --api to also generate an API Controller with Swagger and v1 routes
```

### Expected result

```bash
# Web only (default)
php artisan ptah:forge Product \
  --fields="name:string,sku:string,price:decimal,stock:integer,category_id:unsignedBigInteger,is_active:boolean,description:text" \
  --soft-delete

# Web + API (combined mode — recommended for marketplace/app scenarios)
php artisan ptah:forge Product \
  --fields="name:string,sku:string,price:decimal,stock:integer,category_id:unsignedBigInteger,is_active:boolean,description:text" \
  --soft-delete --api
```

## Prompt: Configure BaseCrud

Use when the entity has already been generated and you need to configure the dynamic table.

```
Configure BaseCrud for the Product entity in the database (crud_configs table).

VISIBLE COLUMNS:
  - name      → Text, sortable, searchable
  - sku       → Text, sortable, primary badge
  - price     → Currency (USD), right-aligned, sortable
  - stock     → Integer, badge: green if >= 5, red if < 5
  - is_active → Boolean with icon: ✅ Active / ❌ Inactive

CREATE/EDIT MODAL:
  - All fields above + description (textarea) + category_id (searchDropdown)
  - Width: md (768px)

QUICK FILTERS:
  - status: All / Active only / Inactive only
  - Quick date filter on updated_at

RULES:
  - Use Ptah standard: Boxicons (bx bx-*) or FontAwesome (fas fa-*) icons
  - Dark mode must work — use ptah-dark classes in conditional CSS
  - Preserve colour tokens: primary=#1e40af, success=#10b981, danger=#ef4444
```

## Prompt: Add Validation and Business Logic

Use to add validations to an existing Livewire component.

```
In the Livewire component Ptah\Livewire\Company\CompanyList, add:

1. VALIDATION: The `email` field must be unique in the ptah_companies table,
   ignoring the company being edited (pattern: Rule::unique()->ignore($editingId)).

2. RULE: Before saving, if is_default changes from false to true,
   remove the is_default flag from all other companies in the table.

3. FEEDBACK: When the swap is complete, fire a Livewire event
   `company-default-changed` so other components on the page
   can react (e.g.: CompanySwitcher updates its visual state).

CONSTRAINTS:
  - Use wire:model.blur on text/email fields (not wire:model.live)
  - Do not add local <style> — CSS goes in forge-dashboard-layout.blade.php
  - Follow the same pattern as the other fields in the existing rules() method
```

## Prompt: Write Tests

Use to request tests aligned with the package test structure.

```
Write tests for the company module of the jonytonet/ptah package.

TEST ENVIRONMENT SETUP:
  - Extends Ptah\Tests\TestCase (Orchestra Testbench + SQLite :memory:)
  - Use CompanyFactory::new()->create([...]) — do NOT use Company::factory()
    (the package does not use native Eloquent Factories)
  - Testbench automatically registers PtahServiceProvider —
    CompanyService is already available via app(CompanyServiceContract::class)

REQUESTED COVERAGE:
  1. Unit → Model: getLabelDisplay(), scopes active/default, auto-slug, soft delete
  2. Feature → Livewire:
     - can render (assertOk)
     - can create a valid company (assertDatabaseHas)
     - name is required (assertHasErrors(['name' => 'required']))
     - unique label blocks duplicate (assertHasErrors(['label']))
     - can edit without changing label (assertHasNoErrors)
     - cannot delete the default company (assertDatabaseHas with deleted_at null)
     - search filters results

FILE LOCATIONS:
  tests/Unit/Models/CompanyModelTest.php
  tests/Feature/Livewire/CompanyListTest.php
```

## Prompt: Customise Excel/PDF Export

Use when you need to personalise exported value formatting or add calculated columns.

```
Customise the BaseCrud Excel/PDF export for the Product model in the jonytonet/ptah package.

CONTEXT:
  - The system uses maatwebsite/excel (Excel) and barryvdh/laravel-dompdf (PDF)
  - The CrudExport class (Ptah\Exports\CrudExport) controls formatting
  - The Blade template (ptah::exports.pdf) controls the PDF layout
  - Only visible columns are exported (action columns excluded)

REQUESTED CUSTOMISATIONS:

1. VALUE FORMATTING:
   - 'money' type fields: display as "$ 1,234.56" (not the raw number)
   - 'enum' type fields: use the label instead of the numeric value
   - Dates: use "Y-m-d" format without time
   - Null fields: display "-" instead of empty

2. CALCULATED COLUMN:
   - Add a "Total" column (quantity * price) in both Excel and PDF
   - This column does not exist in the database — must be calculated during export

3. PDF STYLE:
   - Company logo in the header
   - Roboto font instead of DejaVu Sans
   - Monetary values right-aligned
   - Totaliser in the footer (sum of the 'Total' column)

CONSTRAINTS:
  - Do NOT modify the visible column selection logic
  - Maintain compatibility with CrudConfig (colsNomeLogico, colsTipo, etc.)
  - Use only inline CSS in the PDF template (no Tailwind classes)
  - Formatting must work for both full export and bulk export

FILES TO MODIFY:
  - ptah/src/Exports/CrudExport.php → formatValue() method
  - ptah/resources/views/exports/pdf.blade.php → layout and styles
```

### Expected result

The agent should:

1. Add logic in `formatValue()` for each custom type
2. Create a `getSelectLabel()` helper method if needed
3. Add the calculated column to `map()` and `headings()`
4. Update the PDF template with:
   - Inline `<style>` with Roboto font via Google Fonts
   - Logo in the header (check if it exists first)
   - Conditional alignment per column type
   - Footer with totaliser

## Recommended Workflow

```
1. SCAFFOLDING
   ↓ ptah:forge {Entity} --fields="..." --soft-delete
   ↓ Add --api to generate web + API together (API Controller, Swagger, v1 routes)
   ↓ Use --api-only to generate API only (no web views)
   ↓ Review generated migration, model and DTO

2. BASECRUD CONFIGURATION
   ↓ Build column JSON in the crud_configs table
   ↓ Configure modal, filters and actions

3. BUSINESS LOGIC
   ↓ Service layer — logic in {Entity}Service.php
   ↓ Validations in Requests or Livewire (Rule::unique, etc.)

4. TESTS
   ↓ Unit: model + service
   ↓ Feature: Livewire (create, validate, edit, delete)

5. DOCS
   ↓ Update docs/{Entity}.md if it is a package module
   ↓ Semantic commit: feat: / fix: / docs:
```

## Productivity Tips

### Provide the relevant file

Instead of describing a file, open it in the editor and ask the agent to read the context. The agent analyses the real code before suggesting changes.

```
// Good ✅
"Look at CompanyList.php open in the editor and add unique email validation"

// Less effective ❌
"In the file that handles the company CRUD, add unique email validation"
```

### Use existing code references

```
// Good ✅
"Follow the same Rule::unique pattern already used for the label field in CompanyList::rules()"
```

### Remind the agent of visual constraints

```
// Always include in prompts that touch CSS/visuals:
"CSS is centralised in forge-dashboard-layout.blade.php — do not create local <style>.
 Icons: Boxicons (bx bx-*) or FontAwesome (fas fa-*), never inline SVG."
```

### Long sessions — reset context

In sessions involving many files, summarise the current state for the agent:

```
"Context: jonytonet/ptah package, branch main, commit abc1234.
 We have already implemented X and Y. Now we are implementing Z.
 Relevant files: [list the files]"
```

## Common Pitfalls

| Pitfall | How to avoid |
|---|---|
| Agent uses `Company::factory()` | Always say: "use `CompanyFactory::new()` — there is no Eloquent Factory" |
| CSS in local `<style>` in the view | Remind: "CSS is centralised in `forge-dashboard-layout.blade.php`" |
| `wire:model.live` on text inputs | Remind: "use `wire:model.blur` on text/email/phone fields" |
| Inline SVG as icon | Remind: "use Boxicons CSS classes (`bx bx-*`) or FontAwesome (`fas fa-*`)" |
| Ignoring soft delete in uniqueness | Remind: `Rule::unique()->ignore($id)` and check `withTrashed()` when needed |
| Agent generates manual bind in TestCase | "`PtahServiceProvider` already registers all binds automatically" |
| Commit without semantic message | Pattern: `feat:` / `fix:` / `docs:` / `refactor:` / `test:` |
| UI text hardcoded in the view | "UI strings go in `__('ptah::ui.KEY')` — never hardcode visible strings" |
| Manually filling `created_by`/`updated_by` | Never fill these fields in service/controller/Livewire — the `HasAuditFields` trait and `BaseCrud` do it automatically via Eloquent events |
| Using `->whereIn()->delete()` in bulk | This bypasses Eloquent events — use `->each(fn($r) => $r->delete())` so `HasAuditFields` fires `deleted_by` on each record |

## Performance and High Concurrency

> This project targets **high performance and high concurrency**.
> Performance is a first-class requirement, not an implementation detail.

### Cardinal rules for any code prompt

Always include these constraints when requesting code generation:

```
PERFORMANCE CONSTRAINTS (mandatory):
- NEVER generate foreach inside foreach on Eloquent collections
  Use keyBy() / groupBy() + O(1) key lookup
- NEVER generate a query inside a loop
  Collect IDs first, then use a single whereIn() query
- ALWAYS use with(['relation']) before any foreach that accesses relations
- Any heavy operation (email, PDF, export, external API) must be a Job (Queue)
- Cache with Cache::tags()->remember() for frequently read data
- Every new migration must include index() on filter columns, FKs and order columns
- Do not use ->get() without a limit on large tables — use paginate() or chunk()
```

### Performance-aware prompt template

```
Generate the {name} method in {ProductRepository} following high-performance rules:

1. EAGER LOAD: use with(['category','brand']) — never access relations inside foreach
2. NO N+1: collect all required IDs and fetch with a single whereIn()
3. CACHE: wrap with Cache::tags(['products'])->remember('products:active', 1800, fn()=>...)
4. PAGINATED: return paginate(20) — never ->get() without a limit
5. INDEX: confirm that filtered columns have an index in the migration
6. JOB: if the method is heavy (> 200ms), return void and dispatch a Job
```

### Jobs — agent rule

```
Whenever the agent identifies any of these operations in synchronous code:
  - Sending email / SMS / push notification
  - Generating PDF or Excel
  - Calling an external API (payments, shipping, postal codes, ERP)
  - Image processing (resize, crop, compress)
  - Batch stock update
  - Generating a report or heavy aggregation

The agent MUST move that logic to a Job and dispatch it with dispatch().
Never execute synchronously inside a Request/Response cycle.
```

### Cache — agent rule

```
The agent MUST wrap with Cache::tags()->remember() every query that:
  - Fetches reference tables (species, breeds, categories, services)
  - Lists active catalogue products
  - Aggregates dashboard totals
  - Loads user permissions

Recommended TTLs:
  - Immutable reference tables: 86400 (24h)
  - Product catalogue: 1800 (30min)
  - Dashboard aggregates: 300 (5min)
  - User permissions: until logout (invalidate on login/logout)
```

### Indexes — agent rule

Whenever the agent generates a new migration, it must include indexes on:

```php
// Required in every migration
$table->index('is_active');                          // default boolean filter
$table->index('status');                             // enum/status is almost always filtered
$table->index(['deleted_at', 'is_active']);          // soft-delete + active
$table->index(['created_at']);                       // default ORDER BY

// Composite when the query pattern is known
$table->index(['category_id', 'is_active', 'price']); // catalogue filter
$table->index(['client_id', 'status']);               // orders by client
```

### Supporting tools (install in the project)

```bash
# Production
composer require laravel/horizon                  # Redis queue monitoring
composer require laravel/scout                    # full-text search (Meilisearch)

# Development (--dev)
composer require --dev laravel/telescope          # query, job and cache inspector
composer require --dev itsgoingd/clockwork        # browser devtools profiler
```

> **⚠️ Horizon requires Linux/WSL** — `laravel/horizon` depends on the `ext-pcntl` and `ext-posix` extensions, available only on Unix environments.
> Install Horizon **only on the production server or inside WSL**. On native Windows machines, use `php artisan queue:work` for local development.

### Anti-patterns the agent must REFUSE to generate

```
❌ foreach inside foreach on Eloquent collections
❌ query (findBy, where, all, get) inside any loop
❌ ->with() omitted when the view/code accesses relations
❌ Synchronous Mail::send() or SMS inside a web request
❌ Cache::remember() without tags (impossible to invalidate by group)
❌ ->get() on a table without paginate/limit/chunk
❌ SELECT * in list queries (preferred: ->select(['id','name','status']))
❌ Migration without index on filter or FK columns
❌ Heavy logic in a Livewire property without #[Computed]
❌ Synchronous external HTTP call inside a controller
```
